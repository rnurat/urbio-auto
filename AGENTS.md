AGENTS.md

Философия проекта

Этот проект создаётся как инженерная платформа, а не как набор страниц или отдельных фич.

Основная задача — не просто реализовывать функциональность, а строить масштабируемую, поддерживаемую и расширяемую систему, которая:

* ускоряет разработку,
* улучшает Developer Experience,
* снижает технический долг,
* упрощает развитие проекта в будущем.

Каждое архитектурное решение должно улучшать:

* поддерживаемость,
* масштабируемость,
* консистентность,
* скорость разработки,
* удобство работы команды.

Код не должен писаться ради кода.

Мы создаём инструменты, abstractions и инженерные системы, которые делают проект лучше с каждой новой задачей.

⸻

Инженерные принципы

1. Platform-first подход

Каждая новая фича должна улучшать платформу в целом.

Мы избегаем:

* дублирования логики,
* изолированных решений,
* одноразовых реализаций,
* feature-driven архитектуры.

Мы предпочитаем:

* переиспользуемые примитивы,
* composable-системы,
* расширяемые abstractions,
* единые паттерны,
* предсказуемую архитектуру.

Проект со временем должен становиться проще в разработке, а не сложнее.

⸻

2. Developer Experience — часть продукта

DX рассматривается как полноценная часть архитектуры.

Хорошая инженерная система:

* уменьшает cognitive load,
* снижает количество boilerplate,
* упрощает onboarding,
* автоматически поддерживает консистентность,
* минимизирует вероятность ошибок,
* ускоряет разработку.

Мы оптимизируем:

* локальную разработку,
* структуру проекта,
* типизацию,
* naming,
* tooling,
* automation,
* читаемость кода.

⸻

3. Простота важнее “умного” кода

Предпочтение отдаётся:

* читаемости,
* предсказуемости,
* явному поведению,
* понятной структуре.

Мы избегаем:

* overengineering,
* чрезмерных abstractions,
* “магии”,
* скрытого поведения,
* преждевременной оптимизации.

Лучший код — тот, который легко понять и поддерживать спустя время.

⸻

Архитектура

Atomic Design

Frontend-архитектура строится по принципам Atomic Design.

Структура:

* atoms
* molecules
* organisms
* templates
* pages

Каждый слой имеет чёткую ответственность.

⸻

Atoms

Минимальные UI-примитивы.

Примеры:

* Button
* Input
* Badge
* Typography
* Icon

Atoms должны быть:

* независимыми,
* переиспользуемыми,
* composable,
* максимально универсальными.

⸻

Molecules

Комбинации atoms, формирующие небольшие UI-блоки.

Примеры:

* ThemeToggle
* ContactCard
* NavigationItem
* SearchInput

Molecules не должны содержать сложную бизнес-логику.

⸻

Organisms

Крупные интерфейсные секции.

Примеры:

* HeroSection
* BentoGrid
* NavigationPanel
* ContactSection

Organisms могут управлять состоянием и взаимодействием компонентов.

⸻

Templates

Шаблоны layout-структур без конкретного контента.

Templates отвечают за:

* композицию,
* сетку,
* responsive-поведение,
* spacing.

⸻

Pages

Конкретные страницы приложения.

Pages должны оставаться “тонкими”.

Бизнес-логика и сложность не должны накапливаться внутри страниц.

⸻

Frontend-стандарты

UI System

Проект использует:

* shadcn/ui
* TailwindCSS
* class-variance-authority (CVA)
* design tokens
* semantic spacing system

Интерфейс должен оставаться:

* консистентным,
* доступным,
* composable,
* theme-aware.

⸻

Стилизация

Мы избегаем:

* inline styles,
* случайных spacing values,
* magic numbers,
* неконсистентных размеров.

Мы используем:

* design tokens,
* semantic classes,
* централизованные variants,
* единую spacing-систему.

⸻

Dark / Light mode

Светлая и тёмная темы являются полноценной частью системы.

Интерфейс должен:

* одинаково хорошо выглядеть в обеих темах,
* сохранять читаемость,
* поддерживать accessibility,
* оставаться визуально консистентным.

Theme support — не дополнительная фича, а часть архитектуры.

⸻

Проектирование компонентов

Компоненты должны быть:

Переиспользуемыми

Компонент должен решать общую UI-задачу, а не конкретный единичный кейс.

⸻

Предсказуемыми

Поведение компонента должно быть очевидным по его API.

⸻

Composable

Компоненты должны легко комбинироваться между собой.

⸻

Изолированными

Компоненты не должны иметь скрытых зависимостей.

⸻

Типизированными

Строгая TypeScript-типизация обязательна.

⸻

Управление состоянием

Состояние должно находиться максимально близко к месту использования.

Мы предпочитаем:

* server components,
* derived state,
* composition,
* explicit data flow.

Мы избегаем:

* избыточного global state,
* скрытых мутаций,
* глубокой передачи props.

⸻

Структура проекта

Структура файлов должна отражать архитектуру системы.

Мы избегаем:

* “свалки” файлов,
* неочевидных директорий,
* хаотичного нейминга.

Мы придерживаемся:

* domain-oriented структуры,
* предсказуемого naming,
* discoverable architecture.

⸻

Naming conventions

Naming должен быть:

* явным,
* семантическим,
* предсказуемым,
* searchable.

Мы избегаем:

* сокращений,
* абстрактных названий,
* контекстно-зависимого naming.

Хороший naming уменьшает необходимость документации.

⸻

Производительность

Performance является базовым требованием.

Мы приоритизируем:

* server rendering,
* streaming,
* code splitting,
* оптимизацию assets,
* минимальный client-side JavaScript.

Избегаем ненужной hydration.

⸻

Accessibility

Accessibility является обязательным требованием.

Интерфейс должен поддерживать:

* keyboard navigation,
* screen readers,
* правильный contrast,
* semantic HTML,
* focus states.

⸻

Анимации и взаимодействие

Анимации должны:

* улучшать UX,
* помогать восприятию интерфейса,
* быть естественными,
* оставаться сдержанными.

Мы избегаем декоративных эффектов без функциональной пользы.

⸻

Code Review

Code Review нужен для:

* улучшения архитектуры,
* поддержки консистентности,
* снижения технического долга,
* повышения качества системы.

Review должен оценивать:

* поддерживаемость,
* масштабируемость,
* читаемость,
* DX,
* архитектурные решения.

А не личные предпочтения.

⸻

Tooling

Мы предпочитаем automation вместо ручных процессов.

Платформа должна предоставлять:

* linting,
* formatting,
* type safety,
* component generators,
* CI validation,
* automated checks,
* единый tooling.

Инженерное время должно тратиться на продуктовые задачи, а не на рутину.

⸻

Правила ведения commit-сообщений

Общие принципы

Commit-сообщения являются частью инженерной культуры проекта.

Они должны:

* быть понятными,
* краткими,
* описывать инженерный смысл изменений,
* помогать понимать историю проекта спустя время.

Все commit-сообщения ведутся исключительно на русском языке.

⸻

Формат commit-сообщения

Структура:

[type]: Краткое описание
- Детализация изменений
- Дополнительные изменения
- Важные детали

⸻

Пример

[feat]: Создал многофункциональную кнопку
- Добавил variants для кнопки
- Реализовал поддержку loading-state
- Добавил поддержку dark/light mode

⸻

Типы commit-сообщений

feat

Новая функциональность.

[feat]: Добавил Bento Grid навигацию

⸻

fix

Исправление ошибок.

[fix]: Исправил переполнение карточки на mobile

⸻

refactor

Рефакторинг без изменения бизнес-логики.

[refactor]: Упростил структуру HeroSection

⸻

style

Изменения, связанные со стилями или форматированием.

[style]: Обновил spacing в Bento Grid

⸻

docs

Изменения документации.

[docs]: Добавил описание архитектуры Atomic Design

⸻

chore

Технические изменения и обслуживание проекта.

[chore]: Настроил ESLint и Prettier

⸻

perf

Оптимизация производительности.

[perf]: Оптимизировал hydration карточек

⸻

test

Тесты.

[test]: Добавил тесты для ThemeToggle

⸻

Требования к commit-сообщениям

Заголовок commit

Должен:

* быть коротким,
* описывать результат изменений,
* начинаться с глагола.

⸻

Детализация

Список изменений:

* оформляется через -,
* содержит только ключевые изменения,
* не должен превращаться в changelog.

⸻

Чего следует избегать

Плохо:

[feat]: Обновил проект
[fix]: Исправления
[chore]: Мелкие изменения

⸻

Главный принцип commit-history

История commit-сообщений — часть архитектуры проекта.

Она должна быть:

* чистой,
* последовательной,
* инженерно понятной,
* читаемой спустя время.

⸻

Главный принцип проекта

Каждая новая задача должна улучшать проект.

Система должна постоянно эволюционировать в сторону:

* большей консистентности,
* меньшей сложности,
* лучших abstractions,
* сильного DX,
* ускорения разработки,
* удобства поддержки.
